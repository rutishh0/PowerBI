# Rolls-Royce Data Visualizer — Codebase Architecture

This document provides a comprehensive, detailed breakdown of what the application does, why it does it, and exactly how the underlying code operates. 

---

## 1. Core Objective (The "What" and "Why")

The primary goal of this application is to process, normalize, and visualize highly complex, unstructured, and messy Excel reports generated by Rolls-Royce Civil Aerospace systems. These reports (such as Statements of Account, Invoice Registers, Guarantee Administration files, and Opportunity Trackers) often rely on visual formatting rather than strict data schemas. 

**Why it exists:** Traditional BI tools (like PowerBI) struggle with Excel files that change column orders, use colored rows for subtotals, or scatter metadata across the top 15 rows. This application exists to:
1. Intelligently ingest these files without requiring the user to clean them first.
2. Provide an interactive, "hyperfuturistic" web dashboard to instantly analyze the data.
3. Allow the data to be queried via AI.
4. Export the normalized data into branded, standardized PDF and PPTX reports.

---

## 2. Backend Architecture (Python / Flask)

The backend acts as the data ingestion, parsing, AI routing, and document generation engine. It runs on Flask and maintains state in memory (`_parsed_store`) tied to user sessions, while persisting uploaded files and user credentials in a local SQLite database (`db.py`).

### A. The Parsing Pipeline (`parser_universal.py`)
This is the most critical backend component. Since uploaded files have no guaranteed structure, the parser uses heuristics rather than hardcoded column indices.

*   **File Type Detection**: When a file is uploaded, `_load_workbook()` reads all sheets. `detect_file_type()` scans the first 25 rows of every sheet for specific string keywords (e.g., "lpi rate", "trent 900 guarantee"). It assigns scores and classifies the file into one of five categories:
    1.  `SOA` (Statement of Account)
    2.  `INVOICE_LIST` (Flat register of open items)
    3.  `OPPORTUNITY_TRACKER` (MEA Commercial Optimisation)
    4.  `SHOP_VISIT` (Trent engine shop visit history)
    5.  `SVRG_MASTER` (Guarantee Administration)
*   **Dynamic Column Mapping**: Instead of assuming "Amount" is in Column E, methods like `_map_soa_columns()` scan varying header rows for aliases (e.g., "amount in doc", "balance"). Once a column is claimed by an alias, it maps that integer index to a standardized JSON key.
*   **Data Normalization**: Functions like `_to_float()` and `_to_date()` aggressively clean the data, stripping currency symbols, handling Excel decimal commas, and standardizing dates to `YYYY-MM-DD` strings.
*   **Output**: Regardless of how messy the input Excel file is, the parser guarantees it returns a clean, standardized JSON dictionary containing `metadata`, `totals`, and `items/sections`.

### B. API Routing (`server.py`)
`server.py` is the HTTP traffic controller.
*   **`/api/upload`**: Receives multipart form data (Excel/PPTX/Images). Routes Excel files to `parser_universal.py`. The resulting parsed JSON is saved in memory in `_parsed_store[session_id]`, and the raw file bytes are saved to SQLite via `db.py`.
*   **`/api/export-pdf`**: Accepts JSON configuration detailing which files and sections the user wants to export. It routes `OPPORTUNITY_TRACKER` files to `pdf_export_opp.py` and all other data to `pdf_export.py`.
*   **`/api/config`**: Serves a `FEATURE_FLAGS` dictionary (e.g., `show_ai`, `show_compare`, `show_files`) controlled by backend environment variables, allowing UI elements to be toggled securely without changing frontend code.

### C. PDF Generation (`pdf_export.py` & `pdf_export_opp.py`)
*   Uses the `fpdf2` library to statically draw PDFs.
*   **Logic**: It grabs the parsed, normalized JSON from `_parsed_store`, computes mathematical aggregations, and literally draws rectangles, lines, and text cells at specific `(x, y)` coordinates to form tables, KPI boxes, and headers. `pdf_export_opp.py` includes conditional logic to only draw specific sections (like "Estimation Level breakdown") if they were requested by the frontend.

### D. AI and LLM Integration (`ai_chat.py`)
*   Acts as a proxy mechanism to communicate with external Large Language Models (like OpenAI, Qwen, or Anthropic via OpenRouter).
*   **Context Injection**: When a user asks a question, the backend grabs the parsed JSON of the currently uploaded Excel files, converts it into a minified markdown table or dictionary string, appends it as a "System Prompt", and sends it to the LLM. This grounds the AI in the exact reality of the uploaded data.

## 3. Frontend Architecture (Vanilla JavaScript)

The frontend is a Single Page Application (SPA) built purely in Vanilla JavaScript (no React/Vue). It relies heavily on DOM manipulation and global state.

### A. State & View Controller (`app.js`)
*   **Global State**: Maintains `_filesData` (the JSON response from the server) and `_currentView` (Standard, Executive, Compare, AI, etc.).
*   **View Orchestration**: `_showDashboard()` acts as the main router. Depending on what view is selected, it hides all containers and shows the relevant one. To construct the dashboard, it dynamically invokes `RRVisualizer.renderVisualizer()`.
*   **Interactivity**: Binds drag-and-drop file upload listeners, sidebar accordion toggles, and initializes GSAP (GreenSock) animations for smooth page transitions.

### B. The Render Engine (`visualizer.js` & `components.js`)
*   `visualizer.js` takes the sanitized JSON from the backend and produces raw HTML strings.
*   **Router Logic**: The `renderVisualizer()` function inspects the `file_type` property attached to the JSON and delegates execution to a specific private renderer (e.g., `_renderSOA`, `_renderOpportunityTracker`, `_renderSVRG`).
*   **Filtering Architecture**:
    *   **Global Filters (`_globalFilterBar`)**: Renders a top-level sticky bar with dropdowns. When altered, it triggers a callback that filters the underlying JSON data array, recalculates all aggregations (sums/counts) in JavaScript, and *completely overwrites* the dashboard HTML via `el.innerHTML = newHtml`. 
    *   **Inline Filters (`_filterableDataTable`)**: Attaches `onchange` event listeners to `<select>` elements injected directly into table headers. When triggered, it hides/shows `<tr>` elements in the DOM via CSS `display: none` rather than re-rendering the whole component.
*   **Component Modularity**: Functions like `_kpiCard()` and `_sectionHeader()` generate standard HTML snippets to ensure visual consistency across completely different file types.

### C. Charting (`charts.js`)
*   Acts as a wrapper around the `ApexCharts` library.
*   The frontend feeds normalized arrays of labels (e.g., Customer Names) and series data (e.g., Balances) into these functions, which generate SVG-based interactive donut charts, bar charts, and aging timelines.

### D. UI Feature Modularity
Based on `window.__FEATURE_FLAGS` (injected into `index.html` by Flask using Jinja2), the frontend conditionally executes bindings.
*   **`ai-chat.js` & `secret-chat.js`**: Handle WebSocket/HTTP polling to update chat windows, render typing indicators, and parse Markdown responses returned from `server.py` into HTML.
*   **`files.js`**: Fetches the SQLite database records of previously uploaded files and renders them as downloadable links in a restricted view.

## 4. End-to-End Data Flow Example

To summarize *how* it works, here is the lifecycle of an Excel file:
1.  **User Drop**: User drags `Trent_900_SVRG.xlsx` into the browser. `app.js` catches the Drop event and POSTs standard `multipart/form-data` to `/api/upload`.
2.  **Server Ingest**: `server.py` receives the bytes and passes them to `parser_universal.py`.
3.  **Heuristic Parse**: The parser reads sheets, detects keywords like "Claims Summary", establishes the file is `SVRG_MASTER`, extracts headers, and normalizes rows into a strict JSON dictionary. This JSON is saved to `_parsed_store` and returned to the browser.
4.  **Frontend Route**: `app.js` receives the JSON, updates its `_filesData` object, and runs `_showDashboard()`.
5.  **Dynamic Render**: `visualizer.js` checks the `file_type` (`SVRG_MASTER`), calls `_renderSVRG()`, builds HTML for KPIs, filters, and tables using the JSON, and injects it into the DOM. ApexCharts draws the graphs.
6.  **Interactive Filter**: User selects "Qualified" from a dropdown. `visualizer.js` catches the event, filters the JSON arrays in memory, and re-runs `_renderSVRG()` to redraw the UI.
7.  **Generate Report**: User clicks "Generate PDF". `app.js` collects the active filters and POSTs a config JSON to `/api/export-pdf`.
8.  **PDF Draw**: `server.py` looks up the parsed data in memory, applies the requested filters, and uses `fpdf2` logic to print a customized, branded PDF document, which is streamed back to the browser as a downloadable blob.
